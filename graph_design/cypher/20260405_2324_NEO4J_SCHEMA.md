## 1. Interpreted graph design

The graph models two social situations in which a person gives feedback on a painting. The core design challenge is distinguishing **sarcastic criticism** (Alice, dry tone, eye roll) from **genuine praise** (Henry, enthusiastic tone, awe), even though both utterances carry the same literal meaning ("praise").

The solution uses a two-level signal architecture:

- **Shared vocabulary layer** — `SignalType`, `LiteralMeaning`, `Context`, `Tone`, `Expression`, and `IntendedMeaning` nodes are created once and reused across all situations.
- **Instance layer** — each `Situation` owns its own `SituationSignal` nodes, which point into the shared vocabulary via `INSTANCE_OF`. This isolates per-situation evidence while enabling pattern matching against shared types.
- **Pattern layer** — `Pattern` nodes reference `SignalType` nodes (not `SituationSignal` nodes), so a pattern is defined once and fires for any situation whose signal set covers the required types.

The two channels (literal/observational and inference) are kept structurally separate, making it possible to query either the factual record or the reasoning chain independently.

### Inference channel

| Relationship | From → To | Properties | Purpose |
|---|---|---|---|
| `HAS_SIGNAL` | `Situation → SituationSignal` | — | Scopes a derived signal to its situation |
| `INSTANCE_OF` | `SituationSignal → SignalType` | — | Links instance evidence to shared vocabulary |
| `REQUIRES` | `Pattern → SignalType` | `weight` (float, default `1.0`) | Defines the required signal conjunction |
| `PREDICTS` | `Pattern → IntendedMeaning` | `probability` (float `[0,1]`) | Prediction strength when pattern fires |

## 5. Assumptions and ambiguities

### A. `statementId` chosen as Statement uniqueness key
The spec explicitly forbids a uniqueness constraint on `Statement.text`. `statementId` is used instead (`stmt_congrats`, `stmt_great`). Text is stored as a plain property for readability.

### B. Shared singletons for `Context` and `LiteralMeaning`
Both situations share the same `Context` node (`They are looking at my painting`) and the same `LiteralMeaning` node (`praise`). This is intentional per the spec: these are vocabulary concepts, not per-situation instances. The `SituationSignal → INSTANCE_OF → SignalType` bridge ensures that pattern matching still works correctly without cross-situation contamination.

### C. `Tone` and `Expression` nodes are shared vocabulary
`Tone` and `Expression` nodes (e.g. `dry_tone`, `awe`) are shared across situations, consistent with the spec's vocabulary/evidence separation. The per-situation evidence is captured in `SituationSignal` nodes; the `Tone` and `Expression` nodes serve the literal channel only.

### D. `PREDICTS` edge properties are not updatable via MERGE alone
Because `MERGE` on a relationship matches on all properties or none, the `probability` on `PREDICTS` edges is set at creation time. Re-running the script will not change an existing `probability` value. To update probabilities, use an explicit `SET` after `MATCH`.

### E. `match_mode` and `min_coverage` on Pattern nodes
Both patterns use `match_mode: 'all_required'` and `min_coverage: 1.0`, consistent with the spec's conjunctive semantics. The partial-matching extension is supported by the schema (properties and `weight` on `REQUIRES` edges are present) but no partial-mode patterns are instantiated in this toy graph.

### F. `IntendedMeaning` nodes `indifference` and `compliment` are created but not predicted
The spec lists `indifference` and `compliment` as allowed `IntendedMeaning` values. They are created as vocabulary nodes for completeness and future use, but no pattern in this toy graph predicts them.

### G. `HAS_TONE` and `HAS_EXPRESSION` appear on multiple source types
The spec defines both `Situation → HAS_TONE → Tone` (authoritative, used for matching) and `Statement → HAS_TONE → Tone` (supplementary). Similarly for `HAS_EXPRESSION` on both `Situation` and `Person`. Both sets of edges are created; the authoritative `Situation`-sourced edges are the ones traversed by the pattern-matching pipeline.
